# ohcha-web-design

> このスキルの用途:
> Web サイト（コーポレート・サービスサイト・ EC ・メディア等、回遊型サイト全般）の
> 構築プロセス・ UI/UX 設計・デザインのスキル。サイトの新規制作・リニューアル・改善・
> レビューのあらゆる場面で使用する。
> LP（1 ページ・ 1 目的で CV させるページ）は別物なので ohcha-lp-design を使うこと。
> 優先順位：①サイト制作の手順・全体フローは ohcha 作成の ohcha-ai-site-building を最優先、
> ②文章は ohcha-writing、③トンマナ・配色は ohcha-tone-manner の決定に従う。
> そのうえで本スキルが「わかりやすいサイトデザイン」を規定する：
> 制作プロセス（要件整理→基本設計→ワイヤー→ビジュアル→ルール定義→実装連携）、
> UI 原則（メイン/サブ、インタラクションコスト、シグニファイア）、数値基準、
> ohcha 固有の UI 判断基準（情報密度、余白、導線、レスポンシブ、EC コンポーネント）。
> 「サイトを作って」「コーポレートサイトを設計して」「サイトのワイヤーを作って」
> 「UI を見直して」「サイトのデザインをレビューして」などで必ず使用する。
> 参考文献：桂信『プロセス・オブ・ UI/UX［UI デザイン編］』（翔泳社, 2024）、
> 原田秀司『UI デザインの教科書』、『WEB サイトの配色』を実務向けに再構成。

# ohcha Web サイト構築・デザインスキル

## このスキルの役割

回遊・信頼構築が目的の Web サイトを「わかりやすく」作るための判断基準とプロセス。
UI デザインは感性ではなく**理論 8 割・感性 2 割**の仮説検証。正解はないが不正解はある。
チェックすべき 4 テーマ：**迷わない（設計）／わかりやすい（言葉）／見やすい（見た目）／気持ちよく使える（感情）**。

**LP とは別物**。1 ページで CV させる説得装置は **ohcha-lp-design** を使う。本スキルは複数ページを回遊させ、信頼を積み上げるサイトが対象。

## 優先順位（厳守）

1. **制作手順・全体フロー** → ohcha 作成の **ohcha-ai-site-building**（リサーチ→設計→実装→公開前チェック）
2. **文章** → **ohcha-writing** ／ 3. **トンマナ・配色** → **ohcha-tone-manner**（ブランド初回は必ず確認）
4. そのうえで本スキルが UI/UX ・制作プロセス・デザインの見せ方を規定する

## ohcha 固有の UI 判断基準（必読）

サイトの新規制作・ UI 変更・レビューでは、最初に
[`references/ohcha-ui-preferences.md`](references/ohcha-ui-preferences.md) を全文読む。
これは特定サイトの配色指定ではなく、ohcha が繰り返し重視する**普遍的な判断順序**である。

- 追加する前に、不要な装飾・枠・太字・重複導線を減らす
- 色・太さ・面積は、ブランドの雰囲気より先に情報と行動の優先順位を伝えるために使う
- 操作結果と下位項目は、操作した場所の近くに出す
- PC とスマホを別々に最適化し、ホバー・追従・カラムをそのまま流用しない
- 共通部品の変更は一画面で終わらせず、全ページへ展開して実機確認する
- **見出しの蛍光マーカー風装飾（マーカー下線・マーカー罫線）と、セクション全幅の色帯（帯デザイン）は禁止**（2026-07-18 ohcha 指示）。見出しは太字と余白で立たせ、色は情報ボックス等の面積を絞った要素で使う
- **見出し・小見出し・ラベルの上下に置く短い横棒（装飾バー・アクセントライン・区切りの「ー」）も禁止**（2026-07-18 ohcha 指示）。カード内の短い区切り線も同様。見出しは太字・サイズ・余白だけで立たせる

案件固有の明示指示がある場合はそちらを優先する。BENRI MART 固有の色名・文言・商品構成を、
別案件へそのまま持ち込まない。

## 制作プロセス（7 ステップ）

1. **要件整理**: 「誰のために・何のために」を明文化してから着手。デザイナーの個性ではなくプロダクトのあるべき姿を探す
2. **基本設計**: サイトの「軸」（機能軸・ビジネス軸・情報整理軸・行動軸）を整理し、メニュー構成・画面遷移を固める。**各画面の表示要素をメイン/サブに分類**する（以降すべての判断基準になる）
3. **ワイヤーフレーム**: 構造設計→情報整理→レイアウト→全ページ設計→ライティング見直しの順。ページごとに「目的・ゴール」「ユーザーの感情・思考」を意識し、ファーストビューに重要情報が入るか常に検証
4. **ビジュアルの方向性**: ポジショニングマップ（例:シンプル⇔賑やか × 高級⇔親しみ）で狙う位置を可視化し、ムードボードでトーンをすり合わせ、**複数案を作って比較決定**する（クライアント合意の手戻り防止に有効）。※テイスト・配色の決定自体は ohcha-tone-manner のフローで行う
5. **基本ルールの定義**: カラー・形・文字・インタラクションのルールをガイドライン化**してから**全ページに展開する（一貫性の担保）
6. **最終 UI**: メイン/サブ・優先度・グルーピング・目線誘導（上から下・ Z 型）をページごとに検証。**必ず実機（特に小型スマホ）で確認**する
7. **実装連携**: デザインは「渡して終わり」にしない。実装中に定期レビューし、納品前にデザインと実装の差分を確認する（UI バグは後回しにされがちで、まとめてチェックすると炎上する）

## UI 原則

- **インタラクションコスト最小化**: ユーザーが目的達成までに使う頭と指の労力（クリック数だけでなく、迷う時間・指やマウスの移動距離）を減らす。理想はナビの存在が意識から消えること
- **1 画面 1 目的（ヒックの法則）**: 選択肢が増えるほど意思決定は遅くなる。1 画面に機能を詰め込まず、目的を分解して「この画面でユーザーに取らせたい最優先アクション（メイン導線）」を 1 つに定める。長い入力フォームはステップ分割する
- **浅く広い階層**: 「深く狭い（クリック数が多く 1 画面の情報が少ない）」より「浅く広い（少ないクリックで見渡せる）」階層を優先する
- **現在地の明示**: いまどのページ・タブにいるかを「太さ／色／大きさ／背景色／目印（下線・アイコン）」の 5 手法から**2 つ以上組み合わせて**視覚化する（太字だけでは弱い）
- **ラベリングは排他的に**: ナビやカテゴリの名前は、他のカテゴリと意味が被らない言葉を選ぶ（迷いの最大要因）
- **メイン/サブの原則**: すべての要素をメイン情報とサブ情報に分類し、サイズ・太さ・色の強弱で優先度を視覚化する。全要素が同じ強度＝何も伝わらない
- **グルーピング**: 関連情報をまとめ、グループ間は広い余白・見出しとコンテンツ間は狭い余白で階層を見せる
- **シグニファイア確保**: 押せる要素は押せる見た目に（色・面・影・シェブロン「>」等の誘導記号）。ミニマルにしすぎて「どこが押せるかわからない」は最悪。逆に押せない要素を押せそうに見せない（コントラストで区別）
- **共通概念の活用**: ユーザーが他サイトで慣れたパターン（ヤコブの法則）に従う。奇抜な独自 UI は作家性であって親切ではない
- **ナビはコンテンツ量で段階拡張**: 小規模はヘッダ＋フッタのみ。増えたらグローバルナビ＋ローカルナビ。構造重視のサイト（EC 等）は左ナビ、記事重視（メディア）は右ナビが定石
- **下タブとドロワーの使い分け（アプリ的 UI/スマホ）**: 頻繁に使う主要機能は下タブ（指が届き発見性が高い）、それ以外をドロワーに格納。**重要機能までハンバーガーメニューに隠さない**（発見性が下がりアクセス激減）
- **デバイス適応**: スマホ/タブレットではホバー前提の UI を使わない。Android は端末の戻るボタンと画面内の「戻る」が競合しないよう設計。PC は複数カラム＋ホバー補足を活用できる
- **フッタ活用**: ページ末端は悪影響の少ないエリア。サイトマップ・問い合わせ導線・会社情報を集約
- **パンくずリスト**: 現在地明示と経路移動。深い階層を持つサイトでは迷子（＝離脱）防止に必須
- **タブ**: タブ切り替え＝ユーザーのマインド切り替え。タブごとにレイアウトを変えてよい（無理に統一しない）
- **マイクロインタラクション**: 「気持ちよさ」「感情への働きかけ」「操作の誘導」の 3 役割。**0.5 秒以内**に収める（操作を止める演出は NG）

## 数値基準

- 最小タップ領域: **iOS 44×44pt ／ Android 48×48dp**。タップ要素同士は余白を空ける（誤タップ防止）
- コントラスト比（WCAG AA）: 通常テキスト **4.5:1 以上**、大きい文字 3:1 以上
- 色覚多様性への配慮: 赤系×緑系の隣接配色を避け、色以外（太字・線・ラベル）でも区別する。シミュレーションツールで確認
- レスポンシブは**幅 375px 前後の小型スマホを起点**に設計し、ファーストビューの情報到達を検証してから PC へ拡張
- フォントは OS デフォルト系を基本とし、画面内の文字サイズの種類を絞る。Web 本文 14〜16px
- 文字の 3 要素＝サイズ（優先度）・ウエイト（優先度）・行間（読みやすさ）で強弱を設計する
- 配色は 4 色区分（BASE/KEY/SUB/FONT）。FONT は純黒を避け #111〜#333。詳細はトンマナシートに従う

## マイクロコピー（共通レイヤー）

- ボタンは動詞化（「お問い合わせ」→「お問い合わせする」）
- ボタン文言と遷移先の見出しを一致させる（メッセージマッチ）
- 専門用語・システム用語を使わない。エラーは「状態」でなく「次の行動」を示す
- アイコンだけで意味が伝わらない場合は文字を添える（ターゲットの年齢層が高いほど有効）

## UI 要件定義テンプレート（設計・レビュー時に使う）

```text
# UI 設計要件
## 1. 前提条件
- ターゲットデバイス: [例: スマートフォン優先]
- 画面の目的（ゴール）: [例: 問い合わせフォームに到達させる]

## 2. 構成要素の優先順位
1. メイン導線: [例: 「無料相談を予約する」ボタン]
2. サブ導線: [例: 事例を見る]
3. 必須情報: [例: サービス概要・実績・料金]
4. 付随情報: [例: FAQ ・会社情報]

## 3. レイアウト・コンポーネント指定
- メイン導線はスクロール追従（フローティング）で常時タップ可能に
- 必須情報はファーストビューに配置
- 付随情報は別ページに逃さずアコーディオンで同一画面内に（階層を深くしない）

## 4. デバイス最適化チェック
- タップ領域 44pt/48dp 以上、現在地の明示（2 手法以上）、OS 標準挙動との整合
```

## アンチパターン

- メイン/サブを決めずに画面設計を始める（全部同じ強さで何も伝わらない）
- 重要なナビ・機能をハンバーガーメニューに全部隠す（存在に気づかれない）
- ホバー必須の UI（マウスオーバーで出る説明等）をスマホに持ち込む
- スワイプ削除・ダブルタップ等、OS 標準にない独自操作を必須にする（学習コストで離脱）
- 1 画面に選択肢・機能を詰め込む（ヒックの法則違反。フォームはステップ分割する）
- グルーピングせずパーツを並べる／余白に意図がない
- PC モニタだけで確認し実機確認をしない（小型端末で破綻していることが多い）
- ミニマル志向で押せる場所が消える／低コントラストのボタンが非アクティブに見える
- タップできる要素に誘導記号がなく、ユーザーの行動がそこで止まる
- 慣習を無視した奇抜なナビゲーション・レイアウト
- デザインを開発に渡して終わりにする／最終段階でまとめて UI チェックする
- 見出しに蛍光マーカー風の装飾を敷く／セクションを全幅の色帯（帯デザイン）で塗り分ける（2026-07-18 ohcha 禁止指示）
- 見出し・小見出しの上下に短い横棒（装飾バー・アクセントライン）を敷く（2026-07-18 ohcha 禁止指示）
- 同一画面に同じ機能のボタンを重複配置する
- スマホ UI を PC にそのまま流用（またはその逆）

## 品質ゲート（設計・レビューの合格基準）

1. **1 画面 1 目的**が守られ、各ページのメイン導線（最優先アクション）が明確か
2. デバイス特性（入力手段・画面サイズ・ OS 標準挙動）に適合し、物理的に操作しづらい UI がないか
3. ラベリング・ナビ構造に、迷う要素（重複・曖昧な分類）が残っていないか
4. インタラクションコストを下げる工夫（現在地の明示・共通概念・タップ領域・浅い階層）が入っているか
5. 実機（小型スマホ）でファーストビュー・視認性・タップ感を確認したか
6. 不要な枠線・太字・背景色・ボタン・説明要素を削り、情報密度を下げられているか
7. ヘッダー・フッター・ナビ・カード・フォーム等の共通部品が全ページで同じ幅・位置・挙動か
8. アコーディオン、追従ヘッダー、スライダー、モーダルを実際に操作し、矢印・文言・状態表示が一致しているか
