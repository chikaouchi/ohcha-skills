> このスキルの用途:
> Claude Code 等で作ったサイトの「公開前・公開後レビュー」を、非エンジニア（WEB ディレクター）にも
> わかりやすい形で一気通貫実行するスキル。ソースコード検査・機密情報スキャン・リンク生存確認・
> 実ブラウザでの表示/動作検証・ SEO/OGP 確認・セキュリティ露出チェックを行い、
> 「公開ストップ級 / 要判断 / 改善推奨 / 問題なし」の 4 段階で報告する。

> - 「サイトレビューして」「サイトのレビューをして」「公開前チェックして」
> - 「このサイト公開していい？」「公開して大丈夫か見て」
> - 「サイトの品質チェックして」「リリース前に確認して」
> - フォルダや URL を指定した「レビューして」「チェックして」

# ohcha-site-review — サイト公開前レビュースキル

## 目的

非エンジニアのユーザーが安心してサイトを公開できるよう、技術チェックを代行し、
**ユーザー自身が判断すべきことと、機械的に確認できたこと**を明確に分けて報告する。

## 前提・心構え

- ユーザーは非エンジニアの WEB ディレクター。**専門用語には必ず平易な言い換えを添える**
  （例:「lazy-load（画面外の画像を後から読み込む省エネ設定）」「OGP 画像（SNS にシェアした時に出る画像）」）
- 報告は「結論ファースト」。公開して良いかどうかを最初の 1 文で言い切る
- メールアドレス・電話番号等の個人情報は応答内で必ずマスクする（例: `o***@t***.co.jp`）
- 検出した問題を勝手に修正しない。**報告 → ユーザーの指示を待つ**（レビューと修正は別タスク）

## ワークフロー

### Step 0. 対象の特定

対象が曖昧なら AskUserQuestion で確認する（プロジェクト内の候補フォルダを提示 + 「プロジェクト外/URL」の選択肢）。
公開済み URL とローカルソースの両方がある場合は**両方**を見る（差分・古い公開版の検出のため）。

### Step 1. ソースコード検査（ローカルにある場合）

1. ファイル構成の把握: `index.html` の場所、assets、dist/build フォルダの有無
2. **機密情報スキャン**: API キー・トークン・パスワード・メールアドレス・電話番号・クライアント実名を全ファイルから grep
3. HTML 全文を読む: 誤字脱字、仮テキスト（Lorem ipsum / TODO / XXX / サンプル）、
   事実関係の要確認箇所（数値・経歴・年号・コピーライト）、`lang` 属性、alt 属性、
   `target="_blank"` に `rel="noopener"` があるか
4. メタ情報: title / meta description / **og:image（無いことが多い頻出問題）** / og:url / canonical / favicon
5. 画像の重さ: 1 枚 300KB 超は「要圧縮」として報告。**HTML から参照されていない未使用画像**も確認
6. dist/build とソースの差分（`diff -q`）: 古いビルドの公開を検出
7. デプロイ先の確認: `.git/config` の remote などから公開先（GitHub Pages / Netlify 等）を推定

### Step 2. 公開 URL 検査（公開済み/ステージングがある場合）

WebFetch で以下を確認（curl はプロジェクトによって禁止されているため使わない）:

1. トップページの表示と内容がローカルソースと一致するか
2. **全外部リンクの生存確認**（note 記事・ SNS ・サブドメイン等を 1 本ずつ）
3. `https://ドメイン/.git/config` が見えないか（機密露出チェック。トップページの HTML が返れば安全）
4. www あり / なし、http → https のリダイレクト
5. 404 ページの挙動

### Step 3. 実ブラウザ検証（preview ツール）

ローカルソースがあれば静的サーバーを立てて検証する。

- サーバー起動: `.claude/launch.json` に追加して `preview_start`。
  python3 の http.server はサンドボックスで失敗することがあるので、**Bun の一枚サーバー**が確実:

```ts
// scratchpad/serve-site.ts
const root = "<対象フォルダの絶対パス>";
Bun.serve({
  port: 8940,
  async fetch(req) {
    const url = new URL(req.url);
    let path = decodeURIComponent(url.pathname);
    if (path.endsWith("/")) path += "index.html";
    const file = Bun.file(root + path);
    if (await file.exists()) return new Response(file);
    return new Response("Not Found", { status: 404 });
  },
});
```

検証項目:

1. `preview_resize` で **1280px（PC）と 375px（スマホ）両方**のスクリーンショット → 崩れ確認
2. `preview_console_logs`（level: warn）でエラーゼロを確認
3. インタラクション実測: メニュー開閉・アコーディオン・タブ等を `preview_eval` でクリック → 状態確認
4. 壊れた画像チェック: `[...document.images].filter(i => !i.complete || i.naturalWidth === 0)`
   ※ **`loading="lazy"` の画面外画像は未読込でもここに載る（誤検知）**。
   scrollIntoView して 1.5 秒待ってから再判定すること

### Step 4. 判定と報告

チェックリスト全項目（同梱の `checklist.md` 参照）を以下の 4 段階に振り分けて報告する:

| 区分 | 意味 |
|---|---|
| 🚫 公開ストップ級 | 機密漏れ・権利侵害・重大な誤り。公開前に必ず解消 |
| ⚠️ 要判断 | 技術的には問題ないが、ユーザー本人の判断が必要（連絡先の妥当性・経歴の書き方・掲載可否等） |
| 💡 改善推奨 | 公開したままでも直せる（OGP 画像・ favicon ・画像圧縮・アクセシビリティ等） |
| ✅ 問題なし | 確認して OK だった項目（表で列挙し、安心材料として必ず見せる） |

報告に必ず含めるもの:

- 冒頭 1 文の結論（「公開して OK / 直してから公開」）
- ✅ 項目の一覧表（何を確認したかの証明）
- ユーザー自身の目視が必要な項目のリスト（画像の中身・権利関係・問い合わせ送信テスト）
- 次のアクション提案（修正を一括でやるか等）

ユーザーが「レポートを HTML で」と言った場合や成果物として残したい場合は、
同梱の `report-template.html` をベースに `reports/site-review/` へ出力する。

## 同梱リソース

| ファイル | 用途 |
|---|---|
| `checklist.md` | チェック項目の全リスト（A〜G の 7 カテゴリ）。レビュー時にこれを網羅する |
| `report-template.html` | HTML レポートの雛形（すっきりシンプルなデザイン。findings を差し込んで使う） |

## 注意（落とし穴）

- lazy-load 画像の「壊れ判定」誤検知（Step 3-4 参照）
- `preview_click` はアコーディオンでダブル発火することがある → `preview_eval` の `btn.click()` で再確認
- スクリーンショットのデフォルトビューポートは狭い（モバイル相当）。PC 確認は必ず 1280px にリサイズ
- 個人サイトのレビューでは「所属企業の情報（会社ドメインのメール等）の混在」を要判断として必ず拾う
- レビュー中に見つけた個人情報はチャット・出力ファイルの両方でマスクする
