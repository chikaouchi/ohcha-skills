# ohcha-deploy

> このスキルの用途:
> サイト公開・デプロイ作業のスキル（インフラ AI の役割）。
> 静的サイトの公開環境の準備・ GitHub 連携・独自ドメイン/DNS 設定・ SSL ・
> 公開後の反映確認・更新運用・ロールバックまでを担当する。
> 「サイトを公開して」「公開環境を用意して」「ドメインをつなげて」「DNS 設定して」
> 「本番に反映して」「公開後の確認をして」などで必ず使用する。
> 公開前の品質チェックは ohcha-site-review、実装は ohcha-coding が担当し、
> 本スキルは「世に出す作業」だけを規定する。

# ohcha デプロイ・公開スキル

## このスキルの役割

作ったサイトを安全に公開し、更新が自動で反映される状態を作る。
標準構成は **GitHub リポジトリ → Cloudflare Pages 自動デプロイ**（ohcha-cha.com で実運用済みの型）。

## 公開先の選定定石

| 条件 | 推奨 | 理由 |
|---|---|---|
| 静的サイト＋独自ドメイン（標準） | **Cloudflare Pages** | 無料・自動 SSL ・ CDN ・ DNS まで一元管理 |
| 静的サイト＋とりあえず公開 | GitHub Pages | リポジトリだけで完結。独自ドメインも可 |
| フォーム・ API 等の動的処理が少しある | Cloudflare Pages ＋ Functions | 静的主体を崩さず拡張できる |
| WordPress 等の CMS 必須 | 別途レンタルサーバー | 本スキルの標準外。要件から再検討する |

原則：**ビルド不要の静的構成（ohcha-coding の技術選定）を保てば、公開はこの表で迷わない。**

## 公開フロー（標準 7 ステップ）

1. **公開前チェック**: ohcha-site-review を先に通す（機密情報スキャン・リンク切れ・表示確認）。未実施なら公開作業に入らない
2. **リポジトリ準備**: GitHub にリポジトリ作成（公開サイトでも原則 Private で可）。`.gitignore` に `.env`・`tokens/`・`docs/` 内の内部戦略資料など**公開してはいけないものを必ず登録**（内部ドキュメントは公開ディレクトリと分離しておく）
3. **初回 push**: git-management スキルの流儀で記録・ push（force 系は使わない）
4. **Cloudflare Pages 連携**: ダッシュボードで対象リポジトリを接続。ビルド設定は「なし（静的出力）」、公開ディレクトリを指定。→ `*.pages.dev` の URL で即公開される
5. **独自ドメイン接続**: Pages の Custom Domains にドメインを追加。DNS は Cloudflare 管理なら CNAME が自動作成される。外部レジストラ管理ならネームサーバーを Cloudflare に移すのが最も簡単。SSL は自動発行（反映まで数分〜最大 48 時間）
6. **公開後の反映確認**（下記チェックリスト）
7. **運用開始**: 以後の更新は「push すれば自動反映」。この一文をクライアント/運用者に必ず伝える

## 公開後チェックリスト

- [ ] `https://` で表示され、`http://` と `www` 有無がリダイレクトされるか
- [ ] 全ページを実機スマホで 1 周（キャッシュが古い場合はスーパーリロード）
- [ ] OGP の見え方を確認（X/LINE のシェアデバッガ。キャッシュ更新も実施）
- [ ] フォーム送信テスト（テスト送信 → 受信確認 → テストデータ削除）
- [ ] 404 ページの挙動／存在しない URL の確認
- [ ] Google Search Console にプロパティ登録・サイトマップ送信（SEO 運用するサイトのみ。以後は gsc-report で観測）
- [ ] 検索にまだ出したくない場合は `noindex` が入っているか（逆に本公開時は外したか）

## 更新運用・ロールバック

- 更新は必ず Git 経由（サーバー直編集はしない）。作業の流儀は **git-management** スキルに従う
- 公開がおかしくなったら、Cloudflare Pages の **Deployments から直前デプロイに Rollback**（1 クリックで戻る）。Git の履歴操作より先にこれを使う
- 大きめの変更はプレビューデプロイ（ブランチ）で確認してから本番マージする

## セキュリティ・事故防止

- API キー・トークン・顧客情報・内部戦略ドキュメントを公開リポジトリ/公開ディレクトリに置かない（push 前に必ずスキャン。ohcha-site-review の機密チェックと二重で）
- クライアント案件では、リポジトリとホスティングの**所有権を誰のアカウントにするか**を公開前に必ず合意する（納品後の移管トラブル防止）
- ドメインの更新期限・管理者を記録に残す（docs/ に公開構成メモを 1 枚作る）

## 公開構成メモ（納品ドキュメントの型）

公開したら `docs/deploy-note.md` を必ず残す：
リポジトリ URL ／ホスティング（アカウント所有者）／独自ドメインとレジストラ／DNS 管理場所／
更新方法（push で自動反映）／ロールバック手順／担当連絡先。
引き継ぎ時はこの 1 枚を渡せば完結する状態にする。

## 他スキルとの連携

- 公開前の品質判定: **ohcha-site-review**（公開ストップ級が残っていたら公開しない）
- 実装・構成の前提: **ohcha-coding**（静的構成）／全体フロー: **ohcha-ai-site-building**（Phase 4〜5）
- Git 操作の代行・説明: **git-management**（非エンジニア向けの言葉で）
- 公開後の観測: **gsc-report** / **ga4-report**
