# ohcha-writing

> このスキルの用途:
> ohcha の全アウトプットに適用する文章・コピーライティングの基礎スキル。
> 提案資料・ note ・電子書籍・ Web サイト・ LP ・ Instagram ・ LINE ・メルマガ・ DM など、
> 文章を書く/リライトするあらゆる場面で使用する。
> 「完読される構造設計（主眼と骨子）」「AI っぽさを消す言葉の精度」
> 「Web で行動を促す文言（マイクロコピーの基礎）」の 3 本柱で、
> わかりやすく、読み手に行動してもらえる文章に仕上げる。
> 以下のリクエストで必ず使用する：
> 「文章を書いて」「コピーを書いて」「リライトして」「読みやすくして」「推敲して」
> 「AI っぽさを消して」「note の本文を書いて」「LP の文章を書いて」「ボタンの文言を考えて」
> 「資料の文章を整えて」など。
> 参考文献：唐木元『新しい文章力の教科書』（インプレス, 2015）、
> 野津瑛司『「最強の一言」Web コピーライティング』（standards）を実務向けに再構成。

# ohcha 文章・ライティングスキル

## このスキルの役割

ohcha 名義のすべての文章アウトプットの品質基盤。ゴールはただ一つ、**読者に完読され、行動してもらうこと**。
「うまい文章」ではなく「最後まで読まれて、読み手が動く文章」を作る。
細かいルールはすべてこの目的から逆算した手段であり、意図があれば破ってよい（無自覚な逸脱だけを直す）。

## 大原則

1. **3 層構造**: 文章は「事実 → ロジック → 言葉づかい」の順に下から積み上げる。事実が弱いのに言葉を飾らない
2. **書く前に主眼と骨子**: 主眼（言いたいことを 1 文で）と骨子（要素・順番・軽重）を決めてから書く。骨子が決まれば 8 割完成
3. **サビ頭**: 一番おいしい情報を冒頭に置く。基本構成は「**結論 → 問題提起 → 状況説明 → 付帯情報**」
4. **読者の負担ゼロ**: 余計な言葉を削り、係り受けをシンプルにし、一読して意味がわかる状態にする
5. **完読 > ルール**: すべての判断基準は「読者がストレスなく最後まで読めるか」

## 絶対ルール：絵文字を使わない（サイト・ LP ・資料）

**Web サイト・ LP ・提案資料/スライドの本文・見出し・コピーには、絵文字を一切使わない。** 例外はない。
強調は太字・サイズ・配色・言葉の強さで行う（絵文字・記号での装飾に頼らない）。

- 対象: `ohcha-web-design`（サイト）／`ohcha-lp-design`（LP）／`ohcha-slide-design`（資料）が扱うすべての文章
- **例外**: Instagram の投稿本文・キャプションと LINE の配信本文のみ、絵文字の使用を許可する（ただし乱用しない。詳細は各媒体の運用ルール）
- チェックボタン風の記号やアイコン絵文字（✅・👉・🔥 等）も同様に、サイト・ LP ・資料では使わない

## ワークフロー

### Step 1: 準備（構造シートの作成）

1. 材料（5W1H の事実）を箇条書きでリストアップする
2. 主眼（この文章で伝えたいこと）を 1 つに絞って 1 文で書く
3. 話題を「要素・順番・軽重」で整理する。軽重の基準：**A ＝メインアピール／B ＝基本情報／C ＝付帯情報**
4. **主眼に合わない話題は思い切って捨てる**
5. 長文（note ・ ebook ・提案書）は「章単位＋項目単位」の二重構造シートにする

```markdown
【テーマ（主眼）】
（例: ○○が体験できる、たった 50 名のプレミアムなイベント）

【話題の順番と軽重】
1. （話題） ... B
2. （話題） ... A
3. （話題） ... C
※A: メインアピール、B: 基本情報、C: 付帯情報
```

### Step 2: 執筆

- 構造シートの順に上から文章化する
- 行き詰まったら完成度・言葉づかいを気にせず、**まず最後まで書き切る**ことを優先する

### Step 3: 推敲（このあとの精度ルール＋3 レンズでチェック）

- 修正したら部分確認で済ませず、**必ず冒頭から読み直してリズムを確認**する

## 文の精度ルール（執筆・推敲時チェック）

### 文の構造
- 一文一義。1 文に主張は 1 つだけ。目安 60 字前後。複雑な複文は単文にほぐす
- **主語と述語のねじれ（ドリフト）を直す**: 文を主語・述語に還元して一致を確認する（例外: 「特徴は」「長所は」など属性を問う主語は「〜することだ」と「こと」で受ける）
- 主語と述語、修飾語と被修飾語の距離を近づける。係り受けが 2 通りに読める文は書き直す
- 修飾語は「大きく長い順」に配置する
- **逆接以外の「が」、話し言葉の「で」で文をつながない**。文を 2 つに区切る
- 二重否定は肯定に言い換える

### 言葉を削る
- 「〜という」「〜することができる」などの冗長表現は削る（「取ることができる」→「取れる」）
- 同じ単語・文節・文末表現の連続は **2 連で注意、3 連は不可**。言い換えるか文型を変える
- こそあど言葉は最小限。指示対象が曖昧なら具体的な言葉に置き換える
- 「こと」「もの」「〜化」「〜的」「〜性」の名詞化を削る。助詞を省いた行き過ぎた名詞連結はほぐす
- 体言止めは最小限（※ohcha ルール: サイトコピー・資料の説明文は体言止め禁止、「です・ます」の完全な文で書く）

### 事実と客観性
- 「企画」「作品」「コンテンツ」「施策」などのボンヤリワードは具体的な言葉に置き換える
- 数値化できる情報（日付・数量・価格・割合）は積極的に数字で書く
- 「人気」「豪華」「充実」などの主観形容詞には客観的根拠（数値・事実）を添える
- **濁し言葉（など・とか・〜とのこと）で逃げない**。事実確認をしたうえで断定して言い切る
- 専門用語・業界用語は説明を添えて開く（例外: ファン向けコンテンツで距離を縮めたいときだけ意図的に使う）
- 汎用動詞（する・なる・できる）より狭く強い動詞を選ぶ（淹れる・磨く・組む…）
- 理解していない言葉は一語も書かない（必ず裏取り）。固有名詞・数字・最上表現（「初の」「最大の」）は事実関係を確認する

### 並列・列挙
- 並列する語句は概念のレベルを揃える
- 「〜たり」は必ずセットで使う
- 列挙の助詞（と/や/など）は最初の要素の直後にだけ置く

### 約物・品位
- 「！」による強調は使わない。感嘆は事実の強さで伝える
- かぎかっこは会話文・固有名詞のみ。強調目的で使わない
- 丸かっこの補足は最小限にし、地の文で説明する
- 文頭一語目の直後に読点を打たない
- 漢字が続いて画面が黒く見えるときは、形式名詞・補助動詞をひらがなに開く（「〜する事」→「〜すること」「〜して欲しい」→「〜してほしい」）

## 「AI っぽさ」を消すチェックリスト

生成文が「AI が書いた」と言われる典型要因。納品前に必ず確認する。

- [ ] 同じ語尾が 3 連続していないか（〜です。〜です。〜です。）
- [ ] 同じ文型・同じリズムの文が続いていないか（文の長短に緩急をつける）
- [ ] 「〜できます」「〜が可能です」の可能表現を乱発していないか
- [ ] 「まとめると」「いかがでしたか」「〜と言えるでしょう」の紋切り型で締めていないか
- [ ] 「さまざまな」「多くの」「しっかり」「非常に」の水増し副詞・形容詞が浮いていないか
- [ ] 箇条書きと見出しばかりで、地の文の説得が抜けていないか
- [ ] 抽象論だけで、固有名詞・数字・エピソードが入っているか（オリジナリティは事実に宿る）
- [ ] 誰にでも当てはまる一般論になっていないか（読み手を 1 人に絞って書く）
- [ ] 翻訳調（〜することによって、〜という形で、〜することができるでしょう）が残っていないか
- [ ] 単語・文節・文型・段落・全体の 5 レベルで重複チェックをしたか

## 推敲プロセス（3 レンズ）

提出前に最低 1 周ずつ読み返す。

1. **意味**: ロジックは正しいか。一文一義か。係り受けは一意か。主語と述語はねじれていないか
2. **字面**: 漢字/かなの比率をグレースケールで見る。黒すぎ（漢字過多）も白すぎも直す
3. **語呂**: 音読してリズムが悪い箇所・つっかえる箇所を直す

## Web で行動を促す文言（マイクロコピーの基礎）

ボタン・フォーム・エラーメッセージなど UI 上の短い文言のルール。LP/サイト全体の設計は ohcha-lp-design / ohcha-web-design に従い、文言の書き方は本節を使う。

### ボタンの 3 大 NG と置き換え
1. **「送信」「確認」「ダウンロード」**（システム目線）→ ユーザーが得られるものに変える（「無料ガイド（PDF）を受け取る」「アカウントを作る」）
2. **「詳しくはこちら」** → 何がわかるのかを明記する（「○○の料金を見る」「事例を読む」）
3. **「今すぐ購入」「今すぐ申し込む」**（ハードルが高すぎる）→ 一段低いステップに変える（「カートに入れる」「まずは無料で体験してみる」「料金を確認する」）

### 原則
- **行動結果の明示**: ボタンを見ただけで、押した後に何が起きるか・何が手に入るかがわかる
- **クリックトリガー**: ユーザーの不安（料金・解約・スパム・個人情報）を先回りして払拭する一言をボタンの近くに置く（「100% 無料」「いつでも解約できます」「クレジットカードは必要ありません」「30 日間返金保証」）
- **ポジティブ表現の徹底**: 「〜しないでください」ではなく「〜してください」と、取るべき行動を指示する
- **小学生でもわかる言葉**: システム用語・専門用語を使わない。エラーは「認証エラー」ではなく「パスワードが間違っています。お忘れの場合は再設定へ」のように理由と次の行動を示す

### フォームの文言
- 必須項目は「※」ではなく「**必須**」と書き、項目名の前に置く
- 電話番号・生年月日など個人情報を求めるときは、**なぜ必要かの理由を明記**する（「※本人確認のためにのみ使用します」）
- 不要な入力項目は削る

## 媒体別の適用

| 媒体 | 重点ルール |
|---|---|
| 提案資料・企画書 | サビ頭（結論→意図→内容→予算→スケジュール）。1 スライドの文章も一文一義。詳細は ohcha-slide-design |
| note ・ ebook | 構造シート必須。タイトルは主眼そのもの＋固有名詞始まり（ヒト＞コト＞モノ）。テンプレ感のある定型を避ける。ohcha-note-writing スキルと併用 |
| Web サイト・ LP | 冒頭 13〜15 文字に最重要ワード。具体的数字で訴求。ボタン・フォームは上記マイクロコピー節。全体設計は ohcha-web-design / ohcha-lp-design |
| Instagram ・ LINE | 短くても固有名詞・数字を先頭に。絵文字・！の乱用で品位を下げない。閉じた言葉（ファン向け用語）は対象読者を見極めて意図的に使う |
| DM ・営業文 | 相手固有の情報→課題仮説→低負荷 CTA。sales 系メモ（reports/book-to-skill/01）と併用 |

## アンチパターン

- 骨子を決めずにいきなり書き始める
- 読み手が「自分に関係ある」と感じる前に、書き手の言いたいことを続ける
- 感動を「感動しました」と書く（事実描写で読者に発見させる）
- 濁し言葉・伝聞（など・といった・〜とのこと・〜かもしれません）で言い切りから逃げる
- 翻訳文体（「休みを取ることができるでしょう」→「休みを取れるでしょう」）
- 固有名詞だけ差し替えたテンプレ量産
- システム目線のボタン文言・「※」だけの注釈・理由なき個人情報の要求（Web）

## 品質ゲート（納品前の合格基準）

以下をすべて満たしたときのみ提出する。

1. サビ頭（結論）から始まり、主眼に向かってブレずに構成されているか
2. 一文一義で、主語と述語がかみ合っているか
3. 重複語尾・ダラダラ接続（が/で）・不要な指示語などのノイズが排除されているか
4. 感情的な形容詞ではなく、客観的事実・数字で具体性が担保されているか
5. 「AI っぽさ」チェックリストを通したか
6. （Web の場合）ボタン・フォーム文言が行動結果を明示し、不安を先回りで解消しているか

## 他スキルとの連携

- トーン（文体の人格・言葉遣い）はブランドごとに **ohcha-tone-manner** の決定に従う
- スライドの文章は **ohcha-slide-design**、サイトは **ohcha-web-design**、LP は **ohcha-lp-design** が本スキルの上に乗る
- note 記事の形式・シリーズ設計は **ohcha-note-writing** スキルを優先し、文章品質は本スキルで担保する
