# ohcha-ai-site-building

> このスキルの用途:
> AI エージェント（Claude Code / Codex 等）を「実装担当」、人を「ディレクター」として、
> 小規模サイトをリサーチ→事業/サービス設計→実装→運用設計→公開前チェック→公開後の磨き込みまで
> 一気通貫で構築・改善するためのフルフロースキル。
> 「サイトを作りたい」「AI でサイト構築したい」「公開前チェックをして」「サイトを磨き込みたい」
> 「サイトの運用を設計したい」などのリクエストで使用する。業種を問わず適用可能。

# ai-site-building — AI エージェント活用サイト構築スキル

実案件（一覧・詳細・カテゴリ・ガイド記事を持つ問い合わせ獲得型サイト）を
AI エージェントと構築・公開・改善した実績から抽出した再現手順。
どの業種のサイトでも同じ流れで適用できる。

## 0. 大前提：役割分担

- **AI ＝実装担当**：構造化された実装・一括反映・翻訳/整形・生成・下処理が得意。
- **人（ディレクター）＝品質判断**：画面を見て違和感を言語化し、削る/詰める/揃える判断を下す。
- 「AI に作らせる」ではなく「AI を実装担当として使い、ディレクターが品質判断を行う」体制で進める。
- 人が**必ず**確認するもの：金額・制度などの数字／法務・規約表現／顧客向け返信の仕上げ。
- 生成画像を使う場合は「イメージです」を明記する。

進行はフェーズ順。ただし公開後の磨き込み（Phase 5）だけ独立して呼ばれることも多い。
ユーザーの依頼がどのフェーズかを最初に特定してから着手する。

## Phase 0. リサーチ（3C ＋デザイン）

観点は 3C（顧客/市場・競合・自社）。顧客・自社の整理は Phase 1 の戦略資料に反映されるため、このフェーズの主な作業は**競合調査**と**デザイン参考リサーチ**の 2 つ。

### 競合調査
競合 1 社ごとに次の型で整理する（`docs/competitor-notes.md` として保存）：

| 項目 | 見るポイント |
| --- | --- |
| 内容 | 提供サービスの範囲（本業＋周辺サポート） |
| ターゲット | 誰向けか。自サイトとの重なり |
| 機能 | 検索軸・問い合わせ導線（LINE 等）・サポート機能 |
| トーン | コピーの言葉遣い・デザインの印象 |
| 強み / 弱み | 特に「料金開示の有無」「情報の透明性」は差別化の種になる |

### デザイン参考リサーチ
- 「業態が近い順・応用しやすい順」に並べ、**各参考サイトに「今回への応用」を必ず 1 行書く**（眺めるだけのリストにしない）。
- 参考は同業に限定しない。EC ・宿泊・ブランドサイトなど「見せ方が近い」ものを横断する。
- 決めるのは：写真の扱い（大きく見せるか）／タイポの級差／基調色と差し色／読み物コンテンツの位置づけ。

## Phase 1. 事業・サービス設計（作る前に決める）

このフェーズの成果物は **2 つの戦略資料**。この 2 つがサイトの方向性（構造・技術選定・デザイン・運用）をすべて規定するため、**実装より先に必ず作る**。

### 戦略資料①：事業戦略のまとめ（`docs/business-plan.md`・`docs/service-overview.md`）

1. **体制**：誰が何をやるか。**AI を 1 人の人員として体制表に明記**する（人は判断と関係構築、AI は生成と下処理）。
2. **収益モデル**：収益源をフェーズ順に表化。単価目安×件数で簡易試算し、KPI にする 3 変数（例：相談数×成約率×単価）を決める。
3. **ターゲットとゴール**：セグメント別の具体像。KGI（例：問い合わせ数/月）と中間 KPI（流入・ CTA クリック率・問い合わせ率）。
4. **カスタマージャーニー**：流入→回遊→CV の動線を図にし、各ページの役割を定義する。
5. **法務・表記リスクメモ**：断定表現の禁止リスト（「必ず」「絶対」「最安」等）、業法に触れうる行為の切り分け、「専門家確認が必要な項目」の明示。**公開判断に関わるので必ず作る**。

### 戦略資料②：サイト更新方針（`docs/ops-weekly.md` 等）

「作った後、何をどう更新し続けるか」を設計段階で決める。**更新し続けられないコンテンツは最初から作らない**——この判断がサイト構造を決める。

1. **コンテンツの柱**：定期更新するもの（掲載アイテム・記事・配信）と、作り切りのものを分ける。
2. **更新頻度と時間上限**：「週 1 回・合計 2 時間まで」のように**使える時間の上限から逆算**して業務を削る。
3. **人と AI の分担**：各更新業務について「入力は人が置くだけ→AI が整形→人は確認と投稿」の分担を業務単位で明記。
4. **ネタ帳**：定期コンテンツのネタのストック方法と補充ルール。

→ ①がページ構成とデザインの判断基準に、②がデータ設計（Phase 2）と運用の仕組み（Phase 3）の入力になる。
  例：更新頻度が「週 1 ・人は非エンジニア」なら、CMS 導入ではなく「Sheets ＝マスタ＋ AI ＝変換エンジン」構成が最適解になる。

## Phase 2. サイト設計・実装

### サイト設計・構成案（実装の前に）
- Phase 1 の 2 資料から、**サイトマップ＋各ページの役割・構成案**を文書化してから実装に入る（`docs/service-overview.md` にサイト構造として含めてもよい）。
- 構成案の段階で「一覧・詳細・カテゴリ・記事」などページタイプと、それぞれの CV への役割を確定させる。

### TOP（ファーストビュー）はデザイン AI ×実装 AI の分業
実装 AI が最初に作る TOP はそれっぽさ止まりになりやすい。デザインだけ別の AI に任せる分業が有効：

1. **人**が TOP に載せる要素（メッセージ・導線・掲載コンテンツ）を決定する
2. **実装 AI** に「デザイン生成 AI（ChatGPT 等）へ渡すプロンプト」を作らせる（現行のコード構造・使用要素・**レスポンシブ前提**を含めた指示文）
3. **ChatGPT 等**がデザイン案を出す
4. **実装 AI** がそのデザインを既存コードに**正確に**反映し、崩れを直す

以後の修正は Claude Code / Codex など**そのとき楽なツールでよい**。Phase 5 の指示の型が揃っていれば、ツールをまたいでも品質は保てる。画像生成（ヒーロー・カテゴリ・記事アイキャッチ）も同様に生成 AI へ委譲し、命名・配置・スペックのガイドライン（`docs/image-guidelines.md`）を先に作って従わせる。

### 技術選定の定石（小規模・低コスト・ AI 運用前提）
- **静的書き出し**（例：Next.js `output:"export"`）＋ **Git push→自動デプロイの無料ホスティング**（Cloudflare Pages 等）。サーバー費ゼロ・壊れにくい・ AI が全ファイルを扱える。
- **作りながら公開**：最初から本番 URL に上げて実機・関係者確認を回す。未公開の間は **Basic 認証（ID/パスワード）で保護**する。リリース時は認証を外し、noindex 等の検索ブロックが残っていないか確認したうえで、**sitemap 送信・ Search Console 登録で積極的にインデックスさせる**。
- **共通設定は 1 ファイルに集約**（サイト名・連絡先・ SNS URL 等 → `src/lib/site.ts` 的な場所）。差し替えが一撃になる。
- **掲載データは型付き 1 ファイル**（例：`src/data/items.ts`）で持ち、マスタは Google Sheets / Excel に置く（Phase 3）。
- **ID＝URL は一度公開したら変更しない**。
- **金額は整数で保持し、表示時にフォーマット**。外貨は「約 X 円」併記＋「目安・レート変動あり」を明記。
- **計測は公開と同時に入れる**：GA4 ＋ Search Console を導入し、Phase 1 で決めた KPI（CTA クリック・問い合わせ等）を GA4 イベントとして設定する。週次運用（Phase 3）の KPI 確認と、公開後の磨き込み（Phase 5）の優先順位判断のデータ源になる。

### 開発ルール（AI に守らせる）
- dev サーバー起動中に本番 build を実行しない（キャッシュ破損）。壊れたら `.next` 等は `rm -rf` でなく `mv` で退避。
- 修正のたびに lint / 型チェック / build を通す。
- 事業計画・提案書・運用マニュアル等の**内部資料は `.gitignore` でローカル限定**にし、公開リポジトリに push しない。push 禁止リストをプロジェクトの CLAUDE.md / HANDOFF.md に明記する。

## Phase 3. 運用設計（AI を運用に組み込む）

### スプレッドシート＝CMS、AI＝変換エンジン
- CMS ツールは導入せず、**マスタ＝Google Sheets / Excel、AI がビルド時に読み取り→検証→データファイル再生成**。
- シートは「1 行目＝日本語見出し（必須/任意を色分け）、2 行目＝項目キー、3 行目以降＝データ」の 3 段構成にすると、人にも AI にも読める。
- 変換ルール（列→フィールド・単位変換・ enum マッピング）を文書化しておく。

### 声かけコマンド（最重要）
定型運用は**日常語のコマンドとしてプロジェクトの CLAUDE.md に定義**する。担当エージェントが変わっても同じ言葉で動く。例：

| 声かけ | 中で起きること |
| --- | --- |
| 「新着を反映して」 | 入力フォルダの雑な原稿（外国語・写真そのまま可）→翻訳・整形→マスタ行案→画像リネーム配置→データ再生成→build 確認→**人が確認後に** push |
| 「配信を作って」 | 新着からテンプレ通りの配信文＋サムネ生成。送信は人 |
| 「今週のネタで記事とリールを作って」 | 1 ネタ 3 展開（SEO 記事＋ショート動画台本/mp4＋配信ひとこと版）を一括生成 |
| 「ネタ帳を補充して」 | 検索トレンドからネタ帳へ N 本追加 |

設計原則：
- **入力は雑で OK**（原文のまま・形式バラバラ）→ AI が整形 → **人は確認と投稿だけ**。
- 週次運用は「週 1 回・合計 1.5〜2 時間」など**時間の上限から逆算**して業務を削る。
- 数字（価格・制度）の確認だけは人が原文と突き合わせる工程をフローに固定する。
- 運用マニュアルは Markdown に加えて **HTML 版**（ブラウザで開ける `manual/index.html` 等）も生成する。非エンジニアの運用者・関係者に共有できる形が、仕組みの定着を左右する。
- サイト構築と同時進行で作る（公開後に作ろうとすると熱が冷めて後回しになる）。

## Phase 4. 公開前チェックリスト

**表示品質**：PC/SP 主要ページ確認／文字 14px 以上／薄グレー文字なし／不自然な改行なし／CTA が押せる見た目／クリック範囲が十分／カード・角丸・罫線が過剰でない
**UX**：一覧で比較しやすい／絞り込み中の条件が見える／SP で再絞り込みしやすい／詳細ページ下に次の導線／記事から回遊できる
**技術・公開**：型チェック・本番 build が通る／差分が意図したファイルだけ／機密・不要ファイル混入なし／Basic 認証の解除・ noindex 等の検索ブロックが残っていないか確認（公開時はインデックスさせる）／sitemap 送信・ Search Console 登録／GA4 導入・ KPI イベント計測の動作確認／公開後に反映ページ確認

## Phase 5. 公開後の磨き込み（ディレクションの型）

### 指示の出し方 5 原則
1. **URL と画面種別を明示**（TOP / 一覧 / 詳細 / 記事のどれか）
2. **スクリーンショットで「どこ」を示す**（余白・改行・クリック範囲・ SP は画像必須）
3. **PC/SP の適用範囲を分ける**（「SP のみ」「PC のデフォルトは一覧表示」）
4. **数値・色・文言を具体指定**（`#FFD45A`・ 24px ・「14px 未満禁止」・「gap 2rem を削除」）
5. **「なぜ嫌か」を添える**（「薄くて 50 代に読みづらい」→ AI が同種問題を自分で避けるようになる）

### AI の癖と補正（頻出 5 パターン）
| AI の癖 | ディレクターの補正 |
| --- | --- |
| カード・角丸を多用する | 罫線と余白で区切る。カードは本当に必要な繰り返し要素だけ |
| サブテキストを薄グレーにする | 14px 未満禁止・重要情報は黒に近く。最高齢の読者基準で判断 |
| 主要導線をテキストリンクで済ます | 主要導線はボタン化。`>` 等の記号で「押せる」ことを明示 |
| 余白を取りすぎる | SP は PC より圧縮。padding/gap/margin を個別指定し実機幅で確認 |
| 公開対象外ファイルを混ぜる | 公開前に差分確認・記録対象を明示・未追跡ファイルは含めない |

### 修正依頼テンプレ
```text
対象ページ: <URL>
修正対象: 添付画像の◯◯部分
問題: ◯◯が見づらい / 押せるとわかりづらい / 余白が大きい
修正内容: PC では◯◯、SP では◯◯にしてください。
色は #XXXXXX、フォントサイズは◯px。余白は今の半分に。
終わったら型チェックとビルドを通し、差分を確認してから反映してください。
```

## Phase 6. 引き継ぎ（エージェント・環境をまたぐ場合）

- `HANDOFF.md` を書く：プロジェクト概要／技術構成／**push 禁止ファイル一覧**／声かけコマンド表／未完了タスク／**ユーザーの好み**（表記ルール・色・トーン）／引き継ぎ先での最初のプロンプト（コピペ用）。
- 声かけコマンドは引き継ぎ後も**同じ言葉で動くよう維持**する。
- 環境チェック（依存インストール・認証・ build 確認）を最初のプロンプトに含め、「確認が終わるまでコミット・ push しない」と明記する。
